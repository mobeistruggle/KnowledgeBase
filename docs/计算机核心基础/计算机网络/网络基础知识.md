# 网络基础

> 计算机与计算机之间通信的技术，通过网卡进行通信

## 通信设备

- 网卡：计算机通信的硬件
- 光猫：光信号和电信号相互转换，连接通讯服务商设备箱
- 路由器：从光猫里获取公网IP地址
- 交换机：连接路由器，供多个设备使用
- 光纤：光信号传播
- 网线：电信号传播
- WiFi：电磁波信号传播

## 通信标识

> 通过两者IP地址与子网掩码进行按位与运算确定目标IP地址是否可以通信

- IEEE：分配Mac地址的组织
- IANA：分配IP地址的组织
- IP地址：定位设备所在位置
  - IPV4地址：32位二进制表示，分成四段每段8位，前三段表示网络段，后一位表示主机位
  - IPV6地址：32位十六进制表示，分成八段每段4位
  - 内网网段
    - 10.0.0.0 ~ 10.255.255.255
    - 172.16.0.0 ~ 172.31.255.255
    - 192.168.0.0 ~ 192.168.255.255
  - 本机地址：127.0.0.1
  - 网络地址：主机位全为0
  - 广播地址：主机位全为1
- 域名：简化IP地址，便于用户访问
- Mac地址：定位具体设备
- 子网掩码：一般为255.255.255.0/24
- 端口：定位程序的编号，16位二进制，可用端口1024~49151
- 网关：处理局域网和公网之间通信

## 架构

- C/S：客户端(client) --- 服务端(server)
- B/S：浏览器(browser) --- 服务端

## OSI七层模型

- 前两层可抽象为一层
- 后三层可抽象为一层

|    层级    | 解决的问题 |      作用      |
| :--------: | :--------: | :------------: |
|   物理层   | 怎么传播？ |    信号转换    |
| 数据链路层 | 传播给谁？ |  决定目标设备  |
|   网络层   | 传播到哪？ |  决定目标位置  |
|   传输层   | 传播方式？ |  决定发送方式  |
|   会话层   | 传播时间？ |  决定发送时间  |
|   表示层   | 传博类型？ |  描述文件类型  |
|   应用层   | 传播内容？ | 接收或发送数据 |

## 网络协议

### 物理层

- 水晶头(RJ-45)：用于在以太网上连接网络设备
- ...

### 数据链路层

- Ethernet：用于在局域网上传输数据，工作方式：广播，发给所有局域网设备
- PPP：用于点对点链路上的传输数据
- WiFi：用于在无线局域网上传输数据

### 网络层

- IP：用于确定电脑所在地址并传输数据包
- NAT：局域网IP与公网IP之间转换
  - SNAT：源地址转换
  - DNAT：目标地址转换

- ARP
  - 原理：发数据时，没有Mac地址，广播给局域网所有设备问mac地址，源ip和交换机都缓存mac地址
  - 作用：用于将IP地址转换为Mac地址
  - ARP缓存表：缓存IP地址和Mac地址的对应关系
  - 

- RARP：用于将多个Mac地址转换为IP地址

### 传输层

- TCP：传输控制协议，传输慢安全性高，会进入会话层
- UDP：数据报协议，传输快安全性低，不会进入会话层

### 会话层

- NetBIOS：用于在局域网中实现计算机与计算机之间的通信
- RPC：用于在不同计算机之前进行远程调用
- Socket：用于在不同计算机上的应用程序之前建立通信

### 表示层

- SSL/TLS：用于加密和解密数据，确保数据传输的安全
- MIME：确定数据传输的类型
- XDR：用于在不同类型的计算机系统之间传输数据

### 应用层

- HTTP/HTTPS：超文本传输协议，用于在浏览器和服务器之间传输数据，HTTPS由HTTP+TLS实现
- FTP：文件传输协议
- SMTP：用于电子邮件的传输
- DNS：将域名解析为IP地址，默认端口53
  - 阿里DNS服务器：223.5.5.5  223.6.6.6
  - 腾讯DNS服务器：119.29.29.29
  - 百度DNS服务器：180.76.76.76
  - 谷歌DNS服务器：8.8.8.8
  - 顶级域名服务器：网站后缀
    - a：返回目标域名的IP地址
    - ns：权限下放, 返回下一层DNS服务器的IP地址

- SSH：远程系统安全连接
- TSL：加密通信
- DHTP：动态分配IP地址
- Telnet：远程登录到计算机

## 超文本传输协议

- 请求数据格式：请求行(请求header的第一行)、请求头、请求体

  | 参数名          | 描述                          |
  | --------------- | ----------------------------- |
  | Host            | 目标IP                        |
  | User-Agent      | 浏览器标识                    |
  | origin          | 源IP                          |
  | Accept          | 可以接收的资源类型            |
  | Accept-Language | 可展示到页面上的自然语言      |
  | Accept-Encoding | 支持的压缩类型                |
  | Content-Type    | 请求体数据格式                |
  | Content-Length  | 请求体数据的长度（字节为单位) |

- 响应数据格式：响应行(响应header的第一行)、响应头、响应体

  #### 响应状态码

  | 状态码 | 描述                                              |
  | ------ | ------------------------------------------------- |
  | 1xx    | 临时状态码，表示请求已接收                        |
  | 2xx    | 请求处理完毕                                      |
  | 3xx    | 请求重定向，服务器响应最新url，需要在发送一次请求 |
  | 4xx    | 请求了服务器没有的资源                            |
  | 5xx    | 服务器程序抛出异常                                |

## 通信方式

> 前后端通信方式
>

### 短轮询

> **polling**：以频繁发送请求来完成服务端与客户端的同步

- **优点：**兼容性器，使用简便
- **缺点：**因频繁发送请求，无效请求过多导致浪费网络带宽资源
- **场景：**适应于小规模且实时性不高的项目

### 长轮询

> **long polling**：隔一个较长的时间来进行发送请求

- **优点：**使用简便，实时性良好
- **缺点：**数据更新快时，需要等待漫长的服务端响应
- **实例：**适应于中小规模的应用，有点实时性

### 长连接

- **websocket**：简称ws，实现双向通信
  - **优点**：网络带宽消耗小，TCP连接复用
  - **缺点**：使用复杂，网络情况要好，需要考虑安全性
  - **场景**：高实时性项目，如：聊天项目
- **服务端推送**：简称SSE，实现服务端主动推送消息
  - **优点**：基于HTTP实现，使用方便
  - **缺点**：文本传输效率低，局限于网页端
  - **场景**：消息推送，如：数据库 数据 推送到客户端
  - **步骤**：
    1. 客户端向后端发送带有指定的请求头，如：接收事件流类型、禁用任何的事件缓存
    2. 服务端响应一个带有指定的响应头，如：事件的媒体类型、编码格式
    3. 客户端创建一个EventSource对象，并使用onopen、onmessage和onerror进行处理
    4. 客户端向后端发送关闭请求
- **消息传输协议**：简称mqtt，低延迟 发布-订阅 消息传输协议
  - **优点**：网络带宽花费极少，功率低
  - **缺点**：浏览器无法直接使用，需通过websocket
  - **场景**：物联网（LOT）、带宽有限
